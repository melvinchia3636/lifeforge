# File Storage SDK

**Status**: Planned (not yet implemented)

## Overview

Replace PocketBase's built-in file storage with a pluggable storage SDK that supports
both local filesystem and S3-compatible providers. The SDK provides a unified API for
file save, retrieve, delete, and thumbnail generation, with TypeScript autocomplete
and runtime validation of collection names.

---

## Motivation

### Current State

- **Files stored in PocketBase** as `file`-type fields on records
- **Multer** writes uploads to the `medium/` temporary directory
- **Every module** manually implements the same pattern:
  1. `fs.readFileSync(media.file.path)` - read from disk
  2. `new File([buffer], name)` - create PocketBase File object
  3. `pb.collection('entries').create({ file })` - upload via PocketBase SDK
  4. `fs.unlinkSync(file.path)` - cleanup temp file
- **Thumbnails** are PocketBase schema-defined (`thumbs: ['200x0']`) - auto-generated
  by PB for image uploads, or manually generated by modules (PDF → PNG, EPUB cover)
- **File URLs** are PocketBase-specific: `forgeAPI.getMedia({ collectionId, recordId, fieldId })`
- **No S3 or external storage** support exists
- **No shared abstraction** - each module copy-pastes the same logic

### Problems

1. **Single vendor lock-in** - tied to PocketBase file storage, cannot use S3/R2/MinIO
2. **Manual boilerplate** - `readFileSync`, `new File()`, `unlinkSync` in every handler
3. **No type safety on collection names** - stringly-typed `pb.collection('entries')`
4. **No runtime validation** - invalid collection names silently fail at PocketBase level
5. **No abstraction** - module authors must understand multer internals
6. **Hard to test** - file operations tangled with business logic

## Storage Path Convention

### Key format

```
{module-id}/{collection}/{field}-{uuid}.{ext}
```

Examples:

```
lifeforge--books-library/entries/file-a1b2c3d4.epub
lifeforge--books-library/entries/thumbnail-e5f6g7h8.png
lifeforge--wallet/transactions/receipt-i9j0k1l2.png
lifeforge--moment-vault/entries/file-m3n4o5p6.mp3
lifeforge--scores-library/entries/pdf-q7r8s9t0.pdf
core/backups/database-u1v2w3x4.db
```

### Rationale

- **No record ID needed** - avoids circular dependency (record doesn't exist yet at save time)
- **UUID ensures uniqueness** - no collisions
- **Module prefix auto-inferred** from `createCoreContext`'s `module.id` (e.g., `lifeforge--books-library`)
- **Listable**: all files for a collection = prefix `{module-id}/{collection}/`
- **S3-native**: `/` renders as folder hierarchy in S3 console
- **Local filesystem**: `/` creates real directories for browsing with `find`, `du`, etc.

### Thumbnail storage

```
{original-key}/thumbs/{size}.{ext}
```

Examples:

```
lifeforge--books-library/entries/thumbnail-a1b2.epub/thumbs/200x0.epub
lifeforge--books-library/entries/thumbnail-a1b2.epub/thumbs/0x512.epub
```

## Core API: `core.storage`

### `save(file, options)`

```typescript
import { CleanedSchemas, CollectionKey, FieldKey } from '@lifeforge/pocketbase'

interface SaveOptions<
  TSchemas extends CleanedSchemas,
  TCollection extends CollectionKey<TSchemas> = CollectionKey<TSchemas>
> {
  collection: TCollection // TypeScript autocomplete + runtime validation
  field: FieldKey<TSchemas, TCollection> // autocomplete narrowed by collection + runtime validation
  key?: string // Raw override (escape hatch)
  thumbs?: string[] // e.g. ['200x0', '0x512'] - only for images
  metadata?: Record<string, string> // Arbitrary metadata
}
```

TypeScript infers `TCollection` from the `collection` argument, which narrows `field`
to only the field names defined in that collection's schema - the same pattern used
by `PBService.filter({ field: ... })`:

```typescript
await core.storage.save(file, {
  collection: 'entries', // TCollection inferred as 'entries'
  field: 'thumbnail', // ← autocomplete: 'file' | 'thumbnail' | 'title' | ...
  thumbs: ['200x0']
})
```

**What `save()` does internally:**

1. **Runtime collection validation**: checks `options.collection` exists in schemas - logs warning and returns `null` if invalid
2. **Runtime field validation**: checks `options.field` exists in the collection's Zod schema shape - logs warning and returns `null` if invalid
3. **Key construction**: `{moduleId}/{collection}/{field}-{uuid}.{ext}`
4. **Reads file from disk** (if multer file) or accepts Buffer
5. **Uploads to provider** (local: writes to disk, S3: PutObject)
6. **Generates thumbnails** (if `thumbs` option provided and file is an image - uses `sharp`)
7. **Cleans up temp file** (deletes multer temp file from disk)
8. **Returns `FileReference | null`**

```typescript
interface FileReference {
  key: string
  filename: string
  mimeType: string
  size: number
  thumbs?: ThumbnailInfo[]
}

interface ThumbnailInfo {
  size: string // '200x0', '0x512', '200x200'
  key: string // Full storage key for the thumbnail
  width: number
  height: number
}
```

### `resolveField(existingKey, incomingValue, options)`

Encapsulates the key resolution state machine for update operations. Handles
the four incoming states from the frontend's `FileInput` component and
`convertFormFileFieldData()`:

- `'keep'` → returns `existingKey` unchanged (no-op)
- `'removed'` → deletes the existing file, returns `null`
- `File` / `Multer.File` → deletes the old file (if any), saves the new one, returns the new key
- `undefined` / `null` → returns `existingKey` unchanged

```typescript
resolveField(
  existingKey: string | null,
  incomingValue: 'keep' | 'removed' | File | Express.Multer.File | undefined | null,
  options: SaveOptions<TSchemas, CollectionKey<TSchemas>>
): Promise<string | null>
```

**Example - single field update:**

```typescript
const thumbnailKey = await core.storage.resolveField(
  existing.thumbnailKey,
  media.thumbnail,
  { collection: 'entries', field: 'thumbnail', thumbs: ['200x0'] }
)
```

**Example - multi-field update:**

```typescript
const [fileKey, thumbnailKey] = await Promise.all([
  core.storage.resolveField(existing.fileKey, media.file, {
    collection: 'entries',
    field: 'file'
  }),
  core.storage.resolveField(existing.thumbnailKey, media.thumbnail, {
    collection: 'entries',
    field: 'thumbnail',
    thumbs: ['200x0']
  })
])
await pb.collection('entries').update(id, { ...body, fileKey, thumbnailKey })
```

### `get(key, options?)`

```typescript
interface GetOptions {
  thumb?: string // Return thumbnail instead of original
}

// Returns
interface FileStream {
  stream: Readable
  mimeType: string
  size: number
}
```

Returns `FileStream | null`. On key ownership mismatch, logs a warning and returns `null`.

### `delete(key)`

Deletes the file AND all associated thumbnails (by listing prefix `{key}/thumbs/`).

On key ownership mismatch, logs a warning and returns a no-op (does not throw).

### `getURL(key, options?)`

```typescript
getURL(key: string, options?: { thumb?: string }): string | null
// → /api/files?key=lifeforge--books-library/entries/file-a1b2.epub&thumb=200x0
// On key ownership mismatch, logs a warning and returns null
```

### `exists(key)`

Checks if a file exists at the given key. Returns `boolean`.

On key ownership mismatch, logs a warning and returns `false`.

## Module Migration Examples

### 1. Simple file upload (music - upload MP3)

```typescript
// ─── CURRENT ───
handler: async ({ pb, body, media: { file } }) => {
  const buf = fs.readFileSync(file.path)
  await pb.collection('entries').create({
    ...body,
    file: new File([buf], file.originalname, { type: file.mimetype })
  })
  fs.unlinkSync(file.path)
}

// ─── AFTER ───
handler: async ({ pb, body, media: { file }, core }) => {
  const ref = await core.storage.save(file, {
    collection: 'entries',
    field: 'file'
  })
  await pb.collection('entries').create({
    ...body,
    fileKey: ref.key
  })
}
```

### 2. File with auto-thumbnail (books-library - create entry)

```typescript
// ─── CURRENT ───
handler: async ({
  pb,
  body,
  media: { file },
  core: {
    media: { retrieveMedia, convertPDFToImage }
  }
}) => {
  let thumbnail: File | undefined
  if (file.mimetype === 'application/pdf') {
    thumbnail = await convertPDFToImage(file.path)
  }
  await pb.collection('entries').create({
    ...body,
    ...(await retrieveMedia('file', file)),
    thumbnail,
    read_status: 'unread'
  })
  fs.unlinkSync(file.path)
}

// ─── AFTER ───
handler: async ({ pb, body, media: { file }, core }) => {
  let thumbnailKey: string | null = null
  if (file.mimetype === 'application/pdf') {
    const thumbRef = await core.storage.save(file, {
      collection: 'entries',
      field: 'thumbnail',
      thumbs: ['200x0']
    })
    thumbnailKey = thumbRef.key
  }
  const fileRef = await core.storage.save(file, {
    collection: 'entries',
    field: 'file'
  })
  await pb.collection('entries').create({
    ...body,
    fileKey: fileRef.key,
    thumbnailKey
  })
}
```

### 3. Multiple files (moment-vault - upload several media files)

```typescript
// ─── AFTER ───

  const refs = await Promise.all(
    files.map((file, i) =>
      core.storage.save(file, {
        collection: 'entries',
        field: `file_${i}`
      })
    )
  )
  await pb.collection('entries').create({
    ...body,
    fileKeys: refs.map(r => r.key)
  })
}
```

### 4. Update with "keep" / "removed" pattern

```typescript
// ─── AFTER ───

  const existing = await pb.collection('entries').getOne(id)

    collection: 'entries',
    field: 'file'
```

## Runtime Validation (Defense in Depth)

`FileStorage.save()` validates both `collection` and `field` at runtime against the
module's schemas, even though TypeScript provides compile-time safety. On failure it
logs a warning and returns `null` - the caller is responsible for handling the null
case (e.g., returning a `400 Bad Request` response):

```typescript
// fileStorage.ts
async save(file: MulterFile | Buffer, options: SaveOptions<TSchemas>): Promise<FileReference | null> {
  // 1. Validate collection
  if (!(options.collection in this.schemas)) {
    this.logger.warn(
      `Unknown collection "${String(options.collection)}" for module "${this.moduleId}". ` +
      `Allowed: ${Object.keys(this.schemas).join(', ')}`
    )
    return null
  }

  // 2. Validate field
  const collectionSchema = this.schemas[options.collection] as ZodObject
  const fields = Object.keys(collectionSchema.shape)
  if (!fields.includes(options.field as string)) {
    this.logger.warn(
      `Unknown field "${options.field}" in collection "${String(options.collection)}". ` +
      `Available fields: ${fields.join(', ')}`
    )
    return null
  }

  // ... proceed with save
}
```

This prevents:

- Malformed requests bypassing the type system (e.g., raw HTTP calls)
- Community modules accidentally accessing collections or fields outside their schema
- Future security issues where collection/field names are user-controlled

## Module-Level Access Control

Every method that accepts a key (`get`, `delete`, `getURL`, `exists`) validates
that the key's module prefix matches the calling module's ID. On failure it logs
a warning and returns `null` (or `void` for `delete`, `false` for `exists`) -
the caller is responsible for handling the null case as a graceful no-op.
This prevents one module from reading, deleting, or accessing files owned by
another module.

### Why this is necessary

All keys follow the format `{moduleId}/{collection}/{field}-{uuid}.{ext}`. Since
`moduleId` is baked into every key at save time, the SDK can check ownership on
every key-accepting operation:

```typescript
// fileStorage.ts

private checkKeyOwnership(key: string): boolean {
  if (!key.startsWith(this.moduleId + '/')) {
    this.logger.warn(
      `Key ownership mismatch: "${key}" does not belong to module "${this.moduleId}"`
    )
    return false
  }
  return true
}
```

This guard is applied internally, transparent to the module author:

```typescript
// All of these log a warning and return null/false on key mismatch:
await core.storage.delete(key) // checks key prefix
const stream = await core.storage.get(key) // checks key prefix, returns null
const url = core.storage.getURL(key) // checks key prefix, returns null
const ok = await core.storage.exists(key) // checks key prefix, returns false
```

### What it prevents

- **Accidental cross-module deletion**: a module cannot delete files from another
  module even if it somehow obtains the key - it's a silent no-op with a warning
- **Malicious key injection**: if an attacker crafts a request with a key from a
  different module, the operation is rejected at the SDK level
- **Community module isolation**: third-party modules are sandboxed to their own
  key namespace - they physically cannot touch files outside it

### Why it's transparent to module authors

Module authors never manually construct keys. They get keys from `save()` →
`FileReference.key`, or read them from PocketBase records (`record.fileKey`).
The module ID is auto-inferred at `FileStorage` construction time from
`createCoreContext`'s `module.id`. No extra work needed.

### `save()` is exempt

The `save(collection, field)` method does NOT validate key ownership against
an existing key since it's creating a new one. Its security is handled by:

1. **Collection validation** - `collection` must exist in the module's schemas
2. **Field validation** - `field` must exist in the target collection's schema
3. **Key auto-construction** - the module ID is always prepended by the SDK,
   so the resulting key is guaranteed to start with the calling module's prefix

(See [Runtime Validation](#runtime-validation-defense-in-depth))

## Implementation Order

### Phase 1: Core Package

1. Create `packages/file-storage/` with package.json, tsconfig, vite config
2. Implement `StorageProvider` interface + types
3. Implement `LocalStorageProvider`
4. Implement `S3StorageProvider` (`@aws-sdk/client-s3`)
5. Implement `createProvider()` factory
6. Implement `FileStorage<TSchemas>` class (wraps provider, adds validation + key construction + thumbnails)
7. Implement `serveFile.ts` Express middleware
8. Install `sharp` for thumbnail generation

### Phase 2: Type System Integration

1. Make `CoreContext` generic - `CoreContext<TSchemas = any>` with `storage: FileStorage<TSchemas>`
2. Update `ForgeContext` - `core: CoreContext<TSchemas>`
3. Add `schemas` to forge contract `getValue()` metadata
4. Update `createCoreContext` to accept schemas and create `FileStorage`
5. Update `controllerLogic.ts` to extract schemas from contract and pass to `createCoreContext`

### Phase 3: Backend Endpoint

1. Initialize storage provider singleton in `apps/api/src/index.ts`
2. Replace `GET /media` PocketBase proxy with `GET /api/files` using storage SDK

### Phase 4: Frontend

1. Update `getMediaHelper.ts` - `{ key, thumb }` instead of `{ collectionId, recordId, fieldId, thumb }`
2. Update all client code that calls `forgeAPI.getMedia()`

### Phase 5: Module Migration

1. Migrate `wallet` module (simplest - single receipt field)
2. Migrate `music` module
3. Migrate `moment-vault` module
4. Migrate `scores-library` module
5. Migrate `books-library` module (most complex - EPUB/PDF thumbnails)
6. Update PocketBase schemas (file fields → text fields)
7. Remove `retrieveMedia()` from CoreContext (no longer needed)

### Phase 6: Cleanup

1. Remove `medium/` directory creation from `apps/api/src/index.ts`
2. Consider switching multer from `diskStorage` to `memoryStorage` (no longer need disk temp files)
3. Remove `convertPDFToImage` and `getEpubThumbnail` forwarders from CoreContext (modules import directly)
